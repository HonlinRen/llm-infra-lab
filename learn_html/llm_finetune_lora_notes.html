1. 本次目标

本次实践的目标不是训练一个很强的生产模型，而是跑通大模型微调的核心链路：

  • 使用数据集 llm-wizard/alpaca-gpt4-data-zh。
  • 抽取前 500 条中文指令数据，保存到 ./data/alpaca_500。
  • 使用 Qwen/Qwen2.5-0.5B-Instruct 作为基座模型。
  • 通过 LoRA 训练 adapter，而不是全量更新模型权重。
  • 训练完成后加载 ./qwen-lora-adapter 做推理。
  • 排查 PyTorch CPU 版导致 GPU 没用上的问题，并切换到 CUDA 版 PyTorch。

2. 必学知识点

基座模型和微调模型

基座模型是预训练或指令微调过的原始模型，例如 Qwen/Qwen2.5-0.5B-Instruct。LoRA 微调不会直接改写完整基座模型，而是额外训练一小组低秩矩阵，称为 adapter。

LoRA 是什么

LoRA 的核心思想是冻结大模型大部分参数，只在注意力层等关键模块旁边插入少量可训练参数。这样可以显著降低显存和训练成本。

  • r：LoRA 的秩，越大表达能力越强，但参数更多。
  • lora_alpha：缩放系数，控制 LoRA 更新的强度。
  • target_modules：LoRA 插入到哪些线性层，本次使用 q_proj、k_proj、v_proj、o_proj。
  • lora_dropout：训练时的 dropout，用于缓解过拟合。

为什么 adapter 不是完整模型

训练输出的 adapter_model.safetensors 只有几 MB，因为它只保存 LoRA 参数。推理时必须同时加载基座模型和 adapter：

微调后模型 = Qwen/Qwen2.5-0.5B-Instruct + ./qwen-lora-adapter

SFT 是什么

SFT 即 Supervised Fine-Tuning，监督微调。它使用指令和答案样本，让模型学习按照指定格式回答问题。本次数据字段是 instruction、input、output。

3. 数据准备

本次使用 Hugging Face 数据集 llm-wizard/alpaca-gpt4-data-zh，只取前 500 条做快速实验。

from datasets import load_dataset

保存后的数据可以通过 load_from_disk 读取：

from datasets import load_from_disk

dataset = load_from_disk("./data/alpaca_500")
print(dataset)

本次确认数据共有 500 条，字段为 instruction、input、output。

Prompt 格式

训练时需要把结构化字段拼成模型可以学习的文本格式：

def format_prompt(example):
    instruction = example["instruction"]
    input_text = example.get("input", "")
    output = example["output"]

    if input_text:
        return (
            f"### Instruction:\n{instruction}\n\n"
            f"### Input:\n{input_text}\n\n"
            f"### Response:\n{output}"
        )

    return f"### Instruction:\n{instruction}\n\n### Response:\n{output}"

4. LoRA 微调流程

核心训练脚本位于 finetune/train_lora.py。主要步骤如下：

  1. 加载 tokenizer。
  2. 加载 Qwen2.5-0.5B-Instruct 基座模型。
  3. 读取 ./data/alpaca_500，如果不存在则从 Hugging Face 下载并保存。
  4. 配置 LoRA。
  5. 配置训练参数。
  6. 使用 SFTTrainer 训练。
  7. 保存 adapter 到 ./qwen-lora-adapter。

运行训练

python .\finetune\train_lora.py

关键训练配置

参数 本次值 说明
per_device_train_batch_size 1 单张 GPU 每步处理的样本数。显存够时可以尝试调到 2 或 4。
gradient_accumulation_steps 8 梯度累积步数。等效 batch size 约为 1 * 8 = 8。
learning_rate 2e-4 LoRA 微调常用学习率，比全量微调通常更大。
num_train_epochs 3 数据集完整训练 3 轮。
fp16 有 CUDA 时开启 降低显存占用并提升训练速度。

5. 训练日志怎么看

训练过程中会看到类似日志：

{'loss': '1.852', 'grad_norm': '0.8231', 'learning_rate': '0.0001587',
 'entropy': '1.955', 'num_tokens': '4.461e+04',
 'mean_token_accuracy': '0.5686', 'epoch': '0.64'}
指标 含义 怎么看
loss 训练损失，表示模型预测错误程度。 整体下降是好信号，单个 step 波动正常。
grad_norm 梯度范数，表示参数更新力度。 本次约 0.6 到 1.5，正常。几十、几百或 NaN 才危险。
learning_rate 学习率，控制每次更新步子大小。 本次从约 2e-4 逐步下降，说明调度器在工作。
entropy 输出分布的不确定性。 辅助观察即可，不如 loss 重要。
num_tokens 累计训练过的 token 数。 用于理解训练规模。
mean_token_accuracy token 级预测准确率。 本次从约 0.52 到 0.60，有上升迹象。
epoch 训练轮数进度。 epoch: 3 表示 3 轮训练完成。

6. GPU 是否用起来

一开始训练日志出现过：

UserWarning: 'pin_memory' argument is set as true but no accelerator is found

这表示 PyTorch 没识别到 CUDA。排查后发现安装的是 CPU 版 PyTorch：

torch 2.12.1+cpu
cuda available False
torch cuda build None
device count 0

修复方式

卸载 CPU 版 PyTorch，安装 CUDA 版 PyTorch：

.\.venv\Scripts\python.exe -m pip uninstall -y torch torchvision torchaudio
.\.venv\Scripts\python.exe -m pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu126

验证 CUDA：

.\.venv\Scripts\python.exe -c "import torch; print(torch.__version__); print(torch.cuda.is_available()); print(torch.version.cuda); print(torch.cuda.get_device_name(0))"

nvidia-smi 怎么看

训练真正用上 GPU 时，nvidia-smi 里会看到：

  • python.exe 出现在 Processes 里。
  • 进程 Type 是 C，表示 CUDA Compute。
  • 显存占用明显增加，例如约 4GB。
  • GPU-Util 有持续波动，例如 40% 左右。
任务管理器默认显示的 3D 利用率不一定能反映深度学习负载。判断训练是否用上 GPU，优先看 nvidia-smi 和 PyTorch 的 torch.cuda.is_available()。

7. 使用微调后的模型

训练结束后，adapter 位于：

./qwen-lora-adapter

目录中关键文件包括：

  • adapter_config.json：LoRA 配置。
  • adapter_model.safetensors：LoRA 权重。
  • tokenizer.json、tokenizer_config.json：tokenizer 相关文件。

加载 adapter 推理

import torch
from transformers import AutoTokenizer, AutoModelForCausalLM
from peft import PeftModel

base_model = "Qwen/Qwen2.5-0.5B-Instruct"
adapter_path = "./qwen-lora-adapter"

tokenizer = AutoTokenizer.from_pretrained(adapter_path, trust_remote_code=True)

model = AutoModelForCausalLM.from_pretrained(
    base_model,
    torch_dtype=torch.float16 if torch.cuda.is_available() else torch.float32,
    device_map="auto",
    trust_remote_code=True,
)

model = PeftModel.from_pretrained(model, adapter_path)
model.eval()

question = "保持健康的三个提示。"

messages = [{"role": "user", "content": question}]
text = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True,
)

inputs = tokenizer(text, return_tensors="pt").to(model.device)

with torch.inference_mode():
    outputs = model.generate(
        **inputs,
        max_new_tokens=256,
        do_sample=True,
        temperature=0.7,
        top_p=0.9,
    )

answer_ids = outputs[0][inputs["input_ids"].shape[-1]:]
answer = tokenizer.decode(answer_ids, skip_special_tokens=True)
print(answer)

如何确认不是原始 Qwen

推理脚本中加入：

print(type(model))
print(model.peft_config)
print("active adapters:", model.active_adapters())

如果看到下面信息，就说明已经加载 LoRA：

<class 'peft.peft_model.PeftModelForCausalLM'>
peft_type=<PeftType.LORA: 'LORA'>
base_model_name_or_path='Qwen/Qwen2.5-0.5B-Instruct'
target_modules={'q_proj', 'k_proj', 'v_proj', 'o_proj'}
单看回答内容无法证明是否加载了微调模型，因为原始 Qwen 也能回答常见问题。更可靠的方式是检查模型类型、LoRA 配置和 active adapter。

8. LoRA 合并与独立部署

为什么要合并 LoRA

LoRA 训练完成后默认得到的是 adapter，它只保存增量权重。使用 adapter 推理时，需要先加载基座模型，再加载 LoRA：

Qwen/Qwen2.5-0.5B-Instruct + ./qwen-lora-adapter

合并 LoRA 的目的，是把 adapter 权重写回基座模型权重，生成一个完整模型目录。合并后部署时不再依赖 peft.PeftModel，可以直接用 Transformers 加载：

./qwen-lora-merged
方式 加载方式 适合场景
Adapter 推理 基座模型 + PeftModel.from_pretrained(adapter) 实验、多个 adapter 切换、节省存储空间。
Merged 模型推理 AutoModelForCausalLM.from_pretrained("./qwen-lora-merged") 独立部署、分享完整模型、减少推理加载链路。

合并脚本

本项目使用 finetune/merge_lora.py 完成合并。核心步骤是加载基座模型、加载 adapter，然后调用 merge_and_unload()：

model = AutoModelForCausalLM.from_pretrained(
    args.base_model,
    dtype=parse_dtype(args.dtype),
    device_map=args.device_map,
    trust_remote_code=True,
)

model = PeftModel.from_pretrained(model, str(adapter_path))
merged = model.merge_and_unload()

merged.save_pretrained(str(output_dir), safe_serialization=True)
tokenizer.save_pretrained(str(output_dir))

默认参数已经和本次实践保持一致：

  • 基座模型：Qwen/Qwen2.5-0.5B-Instruct
  • LoRA adapter：./qwen-lora-adapter
  • 合并输出：./qwen-lora-merged

执行合并

python .\finetune\merge_lora.py

成功时会看到类似输出：

Loading base model: Qwen/Qwen2.5-0.5B-Instruct
Loading LoRA adapter: qwen-lora-adapter
Merging LoRA weights into the base model...
Writing model shards: 100%
Merged model saved to: qwen-lora-merged
You can now load it directly with AutoModelForCausalLM.from_pretrained().

合并后目录

合并后的完整模型目录中会包含完整权重和配置：

qwen-lora-merged/
  config.json
  generation_config.json
  model.safetensors
  tokenizer.json
  tokenizer_config.json
  chat_template.jinja
合并后的 model.safetensors 会明显大于 LoRA adapter。adapter 只有几 MB，而 merged 模型接近基座模型完整大小，这是正常现象。

测试合并后模型

合并后的模型可以直接用 finetune/test_merged_lora.py 测试：

python .\finetune\test_merged_lora.py --question "保持健康的三个提示。" --greedy

核心加载方式不再需要 PeftModel：

    model_path,

三个测试脚本的用途

脚本 测试对象 用途
finetune/test_base_qwen.py 原始 Qwen 基座模型 作为 baseline，对比微调前后的回答差异。
finetune/test_lora_adapter.py 基座模型 + LoRA adapter 验证未合并的 LoRA adapter 是否生效。
finetune/test_merged_lora.py 合并后的完整模型 验证 merged 模型能否独立加载和推理。

注意事项

  • 合并时的基座模型必须和训练 adapter 时使用的基座模型一致。
  • 合并输出目录体积较大，通常不要直接提交到普通 Git 仓库。
  • 如果要发布完整模型，建议使用 Hugging Face Hub 或 Git LFS，并确认模型与数据集许可证允许发布。
  • 合并后如果还想继续训练 LoRA，一般重新从基座模型和 adapter 训练链路开始更清晰。

9. 常见问题

现象 原因 处理方式
GPU 显示有占用，但训练很慢 可能只是桌面程序占用显存，PyTorch 没用 CUDA。 运行 torch.cuda.is_available()，确认是否为 True。
torch 2.x + cpu 安装了 CPU 版 PyTorch。 卸载后安装 CUDA 版 PyTorch。
triton not found 缺少 Triton，FLOPs 统计不可用。 一般不影响训练或推理，可以暂时忽略。
torch_dtype is deprecated Transformers 新版本提示参数名变化。 当前不影响运行。后续可以把 torch_dtype 改成 dtype。
不知道是不是微调后的模型 原始模型和微调模型回答可能相似。 打印 type(model) 和 model.peft_config，确认是 PeftModelForCausalLM。
adapter 文件很小 LoRA 只保存少量额外参数。 这是正常现象。推理时必须同时加载基座模型。

下一步学习建议

  • 尝试对比原始 Qwen 和 LoRA 模型在同一问题上的输出差异。
  • 把 do_sample=False 用于确定性对比，减少随机采样影响。
  • 尝试调整 batch_size、learning_rate、epochs，观察 loss 和输出变化。
  • 尝试把 merged 模型接入本地服务接口，例如 FastAPI、OpenAI-compatible API 或 vLLM。
  • 准备更贴近真实业务的数据集，而不是只用通用问答样本。
