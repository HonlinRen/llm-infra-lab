FP16 vs INT4 量化推理实验笔记

围绕显存占用、加载速度、推理速度和回答质量，对比普通 FP16 推理与 bitsandbytes INT4 量化推理，并沉淀可用于面试表达的结论。

1. 一句话结论

INT4 量化的核心价值是用更低精度表示模型权重，显著降低显存占用；代价是加载和推理时可能引入反量化、低比特 kernel 等额外开销，因此不保证一定比 FP16 更快，并且可能带来一定回答质量损失。

显存占用 -67% 3.19GB 到 1.04GB
加载速度 INT4 更慢 2.85s 到 4.52s
推理速度 -21% 53.61 到 42.32 tokens/s
回答质量 无崩坏 但准确性受模型能力限制

2. 背景知识

FP16 推理

  • 权重通常以 16-bit 浮点数存储和计算。
  • 实现简单，GPU 支持成熟，速度通常比较稳定。
  • 显存占用较高，模型越大越容易受显存限制。

INT4 量化推理

  • 权重以 4-bit 形式存储，常见方案包括 NF4 等。
  • 显存占用明显降低，更适合消费级 GPU 或多模型部署。
  • 实际计算时可能需要反量化到 FP16/BF16，速度不一定提升。
本实验使用 Transformers + bitsandbytes 的 4bit 加载方式：load_in_4bit=True、bnb_4bit_quant_type="nf4"、bnb_4bit_compute_dtype=torch.float16。

5. 怎么解释这些现象

为什么 INT4 显存更低？

FP16 每个权重约占 16 bit，而 INT4 每个权重约占 4 bit。虽然实际还会有量化 scale、zero point 或分组元数据等额外开销，但总体权重显存仍会明显下降。本实验里加载后显存从 3.19GB 降到 1.04GB，说明权重量化带来了非常直接的显存收益。

为什么 INT4 不一定更快？

低比特权重减少了显存读取量，但 Transformers + bitsandbytes 的 4bit 推理通常不是全程原生 INT4 矩阵计算。实际推理中可能需要把 4bit 权重反量化到 FP16/BF16 后再参与计算，或者调用特定低比特 kernel。对于 1.7B 这种相对小的模型，反量化和 kernel 开销可能超过省下来的显存带宽收益，因此 tokens/s 反而低于 FP16。

为什么加载速度也可能变慢？

INT4 加载时不仅要读取权重，还要构建量化线性层、读取或生成量化相关元数据，并完成 device map 分配。相比直接加载 FP16 权重，这些步骤会带来额外耗时。本次复跑后 FP16 加载约 2.85s，INT4 约 4.52s。

6. 回答质量分析

两种模式都能输出完整英文回答，没有明显乱码、格式崩坏或重复循环；但它们对 KV Cache 的解释都不够准确。模型把 KV Cache 说成“缓存 previous inference tasks”或“缓存训练数据/常用信息”，这偏离了 LLM 推理里的真实含义。

准确说法：KV Cache 缓存 Transformer 每一层历史 token 的 Key/Value 张量，使自回归 decode 阶段生成下一个 token 时不必重复计算所有历史 token 的 K/V；代价是上下文越长，KV Cache 占用显存越多。
模式 质量观察
FP16 表达更稳定，但把 KV Cache 泛化成“缓存以前推理任务结果”，概念不准确。
INT4 没有明显崩坏，但同样出现概念偏差，并且把 cache 与训练数据访问混在一起。

因此，本次质量结论应写成：INT4 未出现明显灾难性退化，但在事实准确性上 FP16 和 INT4 都受 1.7B 小模型本身能力限制。

7. 实践注意事项

  • 首次运行如果包含模型下载，加载时间会被网络和缓存污染；要复跑一次再记录加载速度。
  • 显存建议同时记录 memory_allocated、memory_reserved 和 max_memory_allocated。
  • 比较生成速度时，输入 prompt、max_new_tokens、采样参数要保持一致。
  • 回答质量不能只看是否能输出，还要检查事实准确性、完整性、是否有幻觉或概念混淆。
  • INT4 更适合显存紧张、模型更大、部署密度更高的场景；小模型单卡测速时不一定更快。

8. 面试表达

可以这样说：

我做过 FP16 和 INT4 的量化推理对比。以 SmolLM2-1.7B-Instruct 为例，INT4 把加载后显存从约 3.19GB 降到 1.04GB，峰值显存从约 3.27GB 降到 1.11GB，显存节省约三分之二。但在这台机器上，INT4 的生成速度从 53.61 tokens/s 降到 42.32 tokens/s，加载也从 2.85s 增加到 4.52s。我的理解是，INT4 主要降低权重显存和访存压力，但 bitsandbytes 4bit 推理存在反量化和 kernel 开销，小模型上不一定能换来速度提升。质量方面没有明显格式崩坏，但准确性仍取决于模型本身能力和量化误差，需要结合任务人工评估。
