KV Cache 开关对比实验笔记

使用 Transformers 对比 use_cache=True 和 use_cache=False 的生成速度，理解大模型自回归推理中最基础、也最关键的优化机制。

一、核心结论

KV Cache 的作用是缓存历史 token 在注意力层中已经计算过的 Key 和 Value。 在生成新 token 时，模型只需要为新 token 计算新的 K/V，并和缓存中的历史 K/V 做注意力计算。

开启 KV Cache 后，decode 阶段可以避免反复重算历史上下文，因此生成速度通常更快。 关闭 KV Cache 后，每生成一个 token 都会重新计算从 prompt 到当前 token 的完整历史序列。

二、背景知识

1. 自回归生成

大语言模型生成文本时通常是自回归过程：先根据 prompt 预测第一个新 token， 再把这个 token 拼回上下文，继续预测下一个 token。这个过程会循环很多次。

2. Prefill 和 Decode

阶段 作用 特点
Prefill 一次性处理用户输入的 prompt 可以并行处理整段输入，主要影响首 token 延迟
Decode 逐个生成新 token 每一步依赖前一步结果，KV Cache 主要优化这个阶段

3. KV Cache 为什么能加速

Transformer 的 attention 会把每个 token 映射成 Query、Key、Value。 在 decode 阶段，历史 token 的 Key 和 Value 已经计算过，并且不会因为新 token 到来而改变。 所以可以把它们缓存起来，后续生成时直接复用。

直觉理解： 开启 KV Cache 就像给历史上下文做了中间结果缓存；关闭 KV Cache 则像每写一个新字， 都重新从文章开头读一遍并重新计算。

三、实验代码

本次实验脚本位于 cache/kv_cache_compare.py，核心对比代码如下：

outputs = model.generate(
    **inputs,
    max_new_tokens=256,
    use_cache=True,
)
outputs = model.generate(
    **inputs,
    max_new_tokens=256,
    use_cache=False,
)

运行命令：

python.exe cache\kv_cache_compare.py --model Qwen/Qwen2.5-0.5B-Instruct

脚本会统计：

  • latency：生成总耗时。
  • tokens/s：每秒生成 token 数。
  • peak_cuda_memory：CUDA 峰值显存。
  • speedup：开启 KV Cache 后的吞吐提升倍数。

四、实测结果

本实验先用 Qwen/Qwen2.5-0.5B-Instruct 做小模型短输出对比， 再用 HuggingFaceTB/SmolLM2-1.7B-Instruct 和更长生成长度放大 KV Cache 的效果。

实验 1：Qwen2.5-0.5B，生成 256 tokens

实验配置：runs=3，max_new_tokens=256，输入长度 51 tokens。

配置 输入 tokens 生成 tokens 平均耗时 tokens/s 峰值显存
use_cache=True 51 256 4.65s 55.03 0.94 GB
use_cache=False 51 256 5.28s 48.47 0.95 GB 
use_cache=True speedup: 1.14x
latency saved per run: 0.63s

可以看到，开启 KV Cache 后，平均耗时从 5.28s 降到 4.65s， 吞吐从 48.47 tokens/s 提升到 55.03 tokens/s。

实验 2：SmolLM2-1.7B，生成 512 tokens

运行命令：

python.exe cache\kv_cache_compare.py --model HuggingFaceTB/SmolLM2-1.7B-Instruct --max-new-tokens 512

实验配置：runs=3，max_new_tokens=512，输入长度 76 tokens。

配置 输入 tokens 生成 tokens 平均耗时 tokens/s 峰值显存
use_cache=True 76 512 9.16s 55.89 3.31 GB
use_cache=False 76 512 25.76s 19.87 3.23 GB 
use_cache=True speedup: 2.81x
latency saved per run: 16.60s

这组结果更清楚地体现了 KV Cache 的收益：开启后平均耗时从 25.76s 降到 9.16s，吞吐从 19.87 tokens/s 提升到 55.89 tokens/s，加速达到 2.81x。

五、结果分析

为什么 Qwen2.5-0.5B 只快 1.14x

  1. 输入上下文较短，只有 51 个 input tokens，重复计算历史 token 的成本还不够大。
  2. 模型较小，Qwen2.5-0.5B 的计算量不高，KV Cache 的优势没有完全放大。
  3. 生成长度是 256，已经能看到差异，但 prompt 更长、生成更多 token 时通常更明显。
  4. 如果在 GPU 上运行，小模型的并行计算效率较高，关闭 cache 的惩罚会被部分掩盖。

为什么 SmolLM2-1.7B 能达到 2.81x

  1. 模型更大，单步 attention 和前向计算成本更高，重复计算历史上下文的代价更明显。
  2. 生成长度增加到 512 tokens，decode 步数更多，KV Cache 在每一步节省的计算会持续累积。
  3. 输入只有 76 tokens，prefill 占比不高，主要耗时来自 decode 阶段，正好命中 KV Cache 的优化重点。
  4. 开启 KV Cache 的显存峰值从 3.23 GB 增加到 3.31 GB，属于用少量显存换取大量计算时间。

可以用更长生成长度进一步放大差异：

python.exe cache\kv_cache_compare.py --model Qwen/Qwen2.5-0.5B-Instruct --max-new-tokens 512

六、实践意义

KV Cache 是大模型推理优化的核心基础之一。vLLM、TensorRT-LLM、SGLang 等推理框架都会围绕 KV Cache 做更进一步的优化，例如分页管理、显存复用、连续批处理和多请求调度。

  • 单请求生成：KV Cache 减少重复计算，提高 decode 吞吐。
  • 长上下文生成：历史 token 越多，缓存带来的收益越明显。
  • 服务化推理：多个请求并发时，KV Cache 的显存管理会直接影响吞吐和稳定性。
一句话总结： use_cache=True 让模型复用历史 token 的 K/V 中间结果； use_cache=False 会在生成过程中重复计算历史上下文，所以更慢。
